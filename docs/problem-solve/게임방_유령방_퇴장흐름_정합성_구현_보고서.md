## 2. 핵심 구현 요약

### 2.1 Leave 멱등화 및 결과 모델 도입

적용 파일:

- `src/main/java/com/kospot/multi/room/application/usecase/LeaveGameRoomUseCase.java`
- `src/main/java/com/kospot/multi/room/application/vo/LeaveRoomResult.java`
- `src/main/java/com/kospot/common/lock/vo/HostAssignmentResult.java`
- `src/main/java/com/kospot/common/lock/strategy/LuaScriptStrategy.java`

핵심 변경:

1) `LeaveGameRoomUseCase.execute(...)` 반환 타입을 `void`에서 `LeaveRoomResult`로 변경

- 상태 코드 도입:
  - `LEFT`
  - `ALREADY_LEFT`
  - `ROOM_NOT_FOUND_CLEANED`
  - `RETRYABLE_FAILURE`
  - `FATAL_FAILURE`

2) effective room id 계산 로직 추가

- 요청 roomId + `member.gameRoomId`를 함께 사용해 정리 대상을 결정
- 둘 다 없으면 예외 대신 성공성 결과(`ALREADY_LEFT`) 반환

3) room 없음/Redis player 없음을 실패가 아닌 정리 성공으로 전환

- DB room 미존재: `member.leaveGameRoom()` + `ROOM_NOT_FOUND_CLEANED`
- Redis player 미존재: `ALREADY_LEFT`

4) Lua/락 결과의 실패 성격 분리

- `HostAssignmentResult`에 `FailureType`(`RETRYABLE`, `FATAL`) 추가
- Lua에서 `Player not found`는 `ALREADY_LEFT`로 매핑
- 알 수 없는 action/파싱 실패는 `FATAL`로 분리

### 2.2 join/create 전환 복원력 강화 (auto-leave 보정)

적용 파일:

- `src/main/java/com/kospot/multi/room/application/service/AutoLeaveRecoveryService.java`
- `src/main/java/com/kospot/multi/room/application/vo/ReconcileResult.java`
- `src/main/java/com/kospot/multi/room/application/usecase/JoinGameRoomUseCase.java`
- `src/main/java/com/kospot/multi/room/application/usecase/CreateGameRoomUseCase.java`

핵심 변경:

1) `AutoLeaveRecoveryService` 신설 (`REQUIRES_NEW`)

- auto-leave 정리를 독립 트랜잭션으로 수행
- 실패 시 recoverable/fatal을 분리
- recoverable이면 `member.gameRoomId`를 강제로 detach 후 진행

2) join/create에서 direct leave 호출 제거

- `JoinGameRoomUseCase`, `CreateGameRoomUseCase`는 reconciler를 통해 선행 정리
- `FATAL_FAILURE`만 차단, recoverable은 흡수

3) Create 경로 NPE 방어 강화

- profile 미존재 시 `equippedMarkerImage` fallback 처리

### 2.3 이벤트 발행 시점 정합화 (AFTER_COMMIT)

적용 파일:

- `src/main/java/com/kospot/multi/room/application/handler/GameRoomEventHandler.java`

핵심 변경:

- `@EventListener` -> `@TransactionalEventListener(phase = AFTER_COMMIT)`
- join/leave 알림이 롤백된 트랜잭션에 대해 선반영되는 문제를 완화

### 2.4 WebSocket 이탈 경로 통합

적용 파일:

- `src/main/java/com/kospot/multi/room/application/service/RoomExitOrchestrator.java`
- `src/main/java/com/kospot/common/websocket/handler/WebSocketEventHandler.java`
- `src/main/java/com/kospot/common/websocket/connection/handler/WebSocketConnectionStateEventHandler.java`
- `src/main/java/com/kospot/common/websocket/session/service/WebSocketSessionService.java`

핵심 변경:

1) `RoomExitOrchestrator` 도입

- leave 호출의 엔트리 포인트를 단일화
- source/reason 기반 로깅 일원화
- retryable/fatal을 결과 모델로 표준화

2) unsubscribe 경로의 room leave 활성화

- 마지막 room 관련 subscription 해제 시에만 leave 수행
- 남은 room subscription이 있으면 skip

3) grace-expired 경로도 오케스트레이터를 통해 동일 정책 적용

### 2.5 정합성 자동 복구 배치 + 부하 최적화

적용 파일:

- `src/main/java/com/kospot/multi/room/infrastructure/scheduler/GameRoomConsistencyCleanupScheduler.java`
- `src/main/java/com/kospot/multi/room/infrastructure/redis/service/GameRoomRedisService.java`
- `src/main/java/com/kospot/multi/room/infrastructure/websocket/service/GameRoomPlayersBroadcaster.java`

- `src/main/java/com/kospot/member/domain/entity/Member.java`
- `src/main/resources/application.yml`

핵심 변경:

1) cleanup scheduler 추가 (feature flag)

- orphan room/ghost room/dangling member 정리
- batch size 제한 + fixed delay 구성

2) Redis room key 탐색 `KEYS` -> `SCAN`

- `getActiveRoomIds()` 추가
- 브로드캐스터도 roomId 기반으로 조회하도록 변경

3) `member(game_room_id)` 인덱스 추가

- dangling/member-room 조회 비용 완화 목적

4) join 경로 debug file append 제거

- `cannotJoinRoom` 내 디스크 I/O 제거로 tail latency 완화

## 3. 설계 의도와 트레이드오프

### 3.1 멱등화에서의 기준

- "정합성 불일치"를 즉시 실패로 보기보다 "정리 성공"으로 해석 가능한 상태를 확대했다.
- 다만 모든 실패를 숨기지 않기 위해 `RETRYABLE_FAILURE`/`FATAL_FAILURE`를 분리했다.

### 3.2 트랜잭션 경계

- auto-leave를 `REQUIRES_NEW`로 분리해 join/create 본 트랜잭션 롤백 전파의 영향도를 낮췄다.
- 이는 정합성 측면에서 유리하나, 호출 경로가 복잡해질 수 있어 로깅 표준화가 필수다.

### 3.3 WS 경로 통합

- unsubscribe/disconnect/grace-expired 경로를 하나의 leave orchestration 정책으로 수렴시켰다.
- 단, `navigate-room`처럼 정상 전환 시나리오는 기존 skip 정책을 유지해 과도한 leave를 방지했다.

### 3.4 부하/쿼리 안정화

- `SCAN` 전환으로 Redis 블로킹 리스크를 줄였다.
- `member.game_room_id` 인덱스로 cleanup 시 DB 조회 비용 튐을 완화했다.

## 5. 요구사항 대비 반영 매트릭스

1) Leave 멱등화: **반영 완료**

- `ALREADY_LEFT`, `ROOM_NOT_FOUND_CLEANED` 등 성공성 상태 반환

2) Join/Create 자동 보정 강화: **반영 완료**

- `AutoLeaveRecoveryService` + `REQUIRES_NEW` + fatal-only 차단

3) WS 이탈 경로 통합: **부분~대부분 반영**

- unsubscribe + grace-expired 통합 완료
- disconnect/navigate 정책은 기존 흐름 유지 + orchestrator 연계 강화

4) 정합성 자동 복구: **1차 반영 완료**

- scheduler + scan + dangling member 정리 반영

5) 관측성(메트릭): **미반영(후속 필요)**

- 계획서 지표(`leave_already_left_count`, `ghost_room_cleaned_count` 등)는 아직 코드에 미구현

## 6. 남은 리스크 및 후속 권장사항

### 6.1 즉시 후속 권장

1) 메트릭 계측 도입

- `LeaveRoomResult.status`, `ReconcileResult.status`, scheduler cleanup 결과를 카운터로 수집

2) 통합/회귀 테스트 확장

- navigate-room 전환 후 join/create 성공 시나리오
- disconnect/unsubscribe 경쟁 상황에서 중복 leave 안전성

3) 운영 롤아웃

- `GAME_ROOM_CONSISTENCY_CLEANUP_ENABLED=false`로 배포 후 단계적 활성화
- batch size/주기 튜닝(운영 Redis/DB 부하 기반)

### 6.2 설계상 체크 포인트

- `GameRoomRedisService.cleanupPlayerSession(...)`가 참조하는 player-session key 생성 경로는 별도 점검 필요
- `getActiveRoomKeys()`는 현재 roomId 반환 alias로 남아 있어, 이름 정리(의미 일치) 리팩토링이 권장됨

- 자동 보정,

를 코드 레벨에 반영했다.
