# 게임방 유령 방/퇴장 정합성 리팩토링 플랜

## 1) 코드베이스 진단 요약

아래는 현재 코드 기준으로 실제 병목/실패 지점을 정리한 결과다.

### A. leave 실패 기준이 과도하고 상위 흐름을 차단함

- `LeaveGameRoomUseCase`에서 방이 없으면 `member.leaveGameRoom()` 호출 후에도 예외를 던진다. (`src/main/java/com/kospot/multi/room/application/usecase/LeaveGameRoomUseCase.java:65`)
- Redis에서 플레이어가 없으면 `HostAssignmentResult.failure("Player not found in room")`로 처리된다. (`src/main/java/com/kospot/multi/room/application/usecase/LeaveGameRoomUseCase.java:109`)
- 락 결과가 실패면 즉시 `GAME_ROOM_OPERATION_IN_PROGRESS` 예외를 던져 `join/create` 선행 auto-leave까지 실패시킨다. (`src/main/java/com/kospot/multi/room/application/usecase/LeaveGameRoomUseCase.java:78`)

### B. join/create가 auto-leave 성공에 강하게 결합됨

- `JoinGameRoomUseCase.autoLeaveIfInOtherRoom`는 leave 예외를 흡수하지 않고 전파한다. (`src/main/java/com/kospot/multi/room/application/usecase/JoinGameRoomUseCase.java:88`)
- `CreateGameRoomUseCase.autoLeaveIfInRoom`도 동일하게 leave 실패 시 생성이 중단된다. (`src/main/java/com/kospot/multi/room/application/usecase/CreateGameRoomUseCase.java:55`)

### C. WS 이탈 경로가 단일 정책으로 수렴하지 않음

- unsubscribe 기반 게임방 leave는 주석 처리되어 사실상 비활성 상태다. (`src/main/java/com/kospot/common/websocket/handler/WebSocketEventHandler.java:98`)
- disconnect에서 `reason=navigate-room`이면 방 leave를 skip한다. (`src/main/java/com/kospot/common/websocket/handler/WebSocketEventHandler.java:167`)
- grace period 만료 이벤트는 leave를 시도하지만 실패를 warn 로그로만 남기고 재보정 경로가 약하다. (`src/main/java/com/kospot/common/websocket/connection/handler/WebSocketConnectionStateEventHandler.java:25`)

### D. DB/Redis 정합성 자동 복구 장치 부족

- `member.gameRoomId`와 Redis 플레이어 해시(`game:room:{id}:players`) 불일치를 요청 진입 시 정리하는 로직이 없다.
- 주기적 orphan/ghost room 정리 스케줄러가 없다(방송 스케줄러는 존재하나 정합성 정리 목적이 아님). (`src/main/java/com/kospot/multi/room/infrastructure/websocket/service/GameRoomPlayersBroadcaster.java:27`)

### E. 세션 정리 보조 키는 제거 로직만 있고 생성 흐름이 없음

- `cleanupPlayerSession()`는 `game:player:{id}:session` 기반 정리를 수행하지만, 동일 키를 저장하는 코드가 현재 확인되지 않는다. (`src/main/java/com/kospot/multi/room/infrastructure/redis/service/GameRoomRedisService.java:334`)
- 결과적으로 기대한 세션 정리 효과가 제한적일 수 있다.

## 2) 리팩토링 목표(구현 관점)

1. **Leave 멱등화**: 방 없음/플레이어 없음/이미 나감은 실패가 아닌 성공성 결과로 반환
2. **Join/Create 복원력 강화**: auto-leave 실패에 덜 민감하고, stale 상태를 선행 정리 후 진행
3. **WS 경로 통합**: unsubscribe/disconnect/grace-expire/navigate 전환이 동일 정책으로 leave 수렴
4. **정합성 자동 복구**: 경량 요청-시점 reconcile + 배치 cleanup으로 유령 방 누적 차단
5. **관측성 확보**: 결과 코드/메트릭/경고 로그 분리로 장애 신호 은닉 방지

## 3) 설계 변경안

## 3-1. Leave 결과 모델 도입

`LeaveGameRoomUseCase`의 반환 타입을 `void`에서 결과 객체로 확장한다.

- 신규 모델 예시: `LeaveRoomResult`
  - `status`: `LEFT`, `ALREADY_LEFT`, `ROOM_NOT_FOUND_CLEANED`, `RECOVERED_WITH_RECONCILE`, `RETRYABLE_FAILURE`, `FATAL_FAILURE`
  - `requestedRoomId`, `effectiveRoomId`, `memberId`, `action(NORMAL_LEAVE/CHANGE_HOST/DELETE_ROOM/NONE)`
  - `warnings`(정합성 보정 메시지)
- 기존 HTTP API는 200 성공 응답 유지 가능(내부적으로 결과만 기록)
- 예외는 **재시도 필요/시스템 오류**에 한해 제한적으로 사용

## 3-2. Leave 멱등 알고리즘(핵심)

`LeaveGameRoomUseCase` 내부를 아래 순서로 재구성한다.

1) `member` 조회 후 `member.gameRoomId`와 요청 `gameRoomId`를 모두 참조해 `effectiveRoomId` 결정
- 둘 다 null이면 `ALREADY_LEFT`
- 요청 room 없음 + member room 존재면 member room 기준 정리 시도

2) DB room 조회
- room 미존재면 `member.leaveGameRoom()`만 반영하고 `ROOM_NOT_FOUND_CLEANED` 반환(예외 금지)

3) Redis leave + host 재할당/방삭제
- 기존 분산락/Lua 전략은 유지
- 단, "player not found"는 실패가 아니라 `ALREADY_LEFT` 계열로 해석
- 마지막 인원 판단 시 Redis empty + DB member 참조 0이면 room 즉시 삭제 보장

4) DB 반영
- `member.leaveGameRoom()`는 항상 최종 보장(이미 null이어도 멱등)
- `CHANGE_HOST`/`DELETE_ROOM`은 결과 코드와 함께 처리

5) 이벤트 발행 규칙
- 실제 상태 변화가 있을 때만 `GameRoomLeaveEvent` 발행
- `ALREADY_LEFT`는 이벤트 스킵 또는 별도 recovery 이벤트(선택)

## 3-3. Lock/Lua 전략 보완 포인트

- `LuaScriptStrategy`의 `Player not found`를 hard failure로 돌리지 않도록 결과 타입 확장
- `HostAssignmentResult.Action`에 `ALREADY_LEFT`(또는 `NO_OP`) 추가 검토
- **분산 락 보장 범위는 유지**하고 의미만 완화

## 4) Join/Create 자동 보정 리팩토링

## 4-1. 공통 선행 보정 컴포넌트 신설

`RoomMembershipReconciler`(신규) 또는 `AutoLeaveRecoveryService`로 공통화:

- 입력: `memberId`, `targetRoomId(optional)`
- 수행:
  - stale `member.gameRoomId` 감지
  - 필요 시 멱등 leave 호출
  - DB/Redis 불일치 보정
- 출력: `ReconcileResult`(`clean`, `recovered`, `warning`, `failureType`)

`JoinGameRoomUseCase`/`CreateGameRoomUseCase`에서 직접 leave 호출 대신 위 컴포넌트 사용.

## 4-2. 차단 정책 변경

- 기존: auto-leave에서 예외 발생하면 join/create 중단
- 변경: **복구 가능한 실패는 흡수하고 진행**, 최종 진입 직전 재검증만 강제
- 진짜 차단 조건:
  - 방 자체가 만원/비밀번호 오류/게임 진행 중 등 비즈니스 불가 사유
  - 시스템 장애(DB/Redis 공통 장애 등)

## 5) WebSocket 이탈 경로 통합 플랜

## 5-1. 단일 leave 트리거 서비스로 수렴

`WebSocketEventHandler`, `WebSocketConnectionStateEventHandler`가 각자 leave를 직접 호출하지 말고,
`RoomExitOrchestrator`(신규)로 통합:

- `requestExit(memberId, roomId, reason, source, withGrace)`
- source: `UNSUBSCRIBE`, `DISCONNECT`, `GRACE_EXPIRED`, `HTTP`
- 내부적으로 멱등 leave 호출 + 결과코드 로깅/메트릭 집계

## 5-2. unsubscribe 경로 활성화

- 현재 주석 처리된 게임방 unsubscribe leave 로직을 복구
- 단일 채널 unsubscribe만으로 즉시 leave하지 않고,
  - 해당 세션이 동일 room 관련 구독을 더 가지고 있는지 확인 후
  - 마지막 room 구독 해제 시에만 leave 트리거

## 5-3. navigate-room 정책

- `navigate-room`은 즉시 leave skip 가능하되, 다음 진입(join/create/subscribe) 시점에 반드시 reconcile 수행
- grace 만료 후에도 stale room이 남아 있으면 배치/요청 진입 reconcile에서 제거

---

## 6) 정합성 자동 복구(요청 시점 + 배치)

## 6-1. 요청 시점 경량 reconcile

적용 위치:

- `JoinGameRoomUseCase.executeV1(...)`
- `CreateGameRoomUseCase.execute(...)`
- 필요 시 `GameRoomController.leaveRoom(...)` 진입 전후

검사 규칙:

- `member.gameRoomId != null`인데 DB room 미존재 -> member roomId null로 정리
- `member.gameRoomId != null`인데 Redis 플레이어 미존재 -> 멱등 leave 또는 직접 정리
- Redis room 해시에 플레이어 0명 + DB에서 해당 room 참조 멤버 0명 -> room 삭제

## 6-2. 배치성 orphan cleanup (feature flag)

신규 스케줄러(예: `GameRoomConsistencyCleanupScheduler`):

- 주기: 초기 5~10분, 점진 조정
- 대상 제한:
  - 최근 N분 이상 갱신 없는 room key
  - 일정 수(batch size)만 처리
- 작업:
  - ghost room 삭제
  - dangling member.gameRoomId 정리
  - cleanup 결과 메트릭/로그 적재

---

## 7) 관측성(메트릭/로그/알람)

최소 지표:

- `leave_already_left_count`
- `auto_leave_recovery_count`
- `ghost_room_cleaned_count`
- `join_blocked_by_stale_room_count`

로그 표준화:

- `memberId`, `requestedRoomId`, `effectiveRoomId`, `source`, `reason`, `resultStatus`, `lockStrategy`, `elapsedMs`
- `ALREADY_LEFT`는 info, `RECOVERED_*`는 warn(샘플링), `FATAL_*`는 error

## 8) 테스트 계획

## 8-1. 단위 테스트

- Leave 결과 코드별 테스트
  - room 없음 + member.roomId 존재 -> `ROOM_NOT_FOUND_CLEANED`
  - Redis player 없음 -> `ALREADY_LEFT`
  - 정상 leave/change-host/delete-room
- Join/Create 보정 테스트
  - stale room 상태에서도 join/create 성공

## 8-2. 통합 테스트

- 마지막 인원 leave 시 DB room + Redis room key 즉시 제거
- DB/Redis 불일치 조합(4~6패턴) 자동 보정
- `navigate-room -> reconnect -> join/create` 시 유령 방 미발생
- unsubscribe/disconnect/grace-expire 각각에서 동일 결과로 수렴

## 8-3. 동시성/회귀 테스트

- 기존 동시 leave 테스트 유지 + 멱등 재호출 시나리오 추가
- join/create 동시 진입 시 stale 정리 경쟁 조건 검증
- 다중 노드(가능 시)에서 lock strategy 일관성 확인

## 9) 단계별 실행 순서

### Phase 1 (최우선): leave 멱등화 + auto-leave 복원력

1. `RoomExitOrchestrator` 도입
2. unsubscribe leave 경로 복구 및 중복 구독 안전장치 추가
3. navigate-room + grace 정책 정리

### Phase 3: 정합성 배치 + 메트릭

1. cleanup scheduler(FF) 배포
2. 메트릭/로그 대시보드 구성
3. 샘플링/부하 튜닝

## 11) 리스크와 완화

- 리스크: "성공성 완화"로 진짜 오류가 가려질 수 있음
  - 완화: 결과 코드를 실패/복구/정상으로 분리하고 알람 임계치 별도 운영
- 리스크: reconcile 배치가 Redis/DB 부하 유발
  - 완화: feature flag, batch size, 처리 시간 상한, 야간 점진 적용
- 리스크: WS 경로 통합 시 중복 leave 트리거
  - 완화: 멱등 leave 보장 + source/reason 기반 중복 억제 키(짧은 TTL)

## 12) 완료 판정 기준

- 마지막 인원 leave 후 DB room + Redis room key 잔존율 0%
- stale `gameRoomId`가 있어도 join/create 성공률 99.9%+
- leave 요청 중 `ALREADY_LEFT`는 실패로 집계되지 않음
- `join_blocked_by_stale_room_count`가 0으로 수렴
- 회귀/동시성 테스트 통과 + 기존 락 전략 보장 유지

## 13) 사이드 이펙트/부하(쿼리) 재점검 결과

이번 이슈 기준으로, 실제 운영 부작용 가능성이 큰 지점을 우선순위별로 재평가했다.

### P0 (즉시 보정 필요)

1. **트랜잭션 롤백과 Redis/Event 불일치 가능성**
   - `JoinGameRoomUseCase`/`CreateGameRoomUseCase`는 `@Transactional`이며, 내부에서 `autoLeave -> leave`를 호출한다.
   - 이후 join/create 단계에서 예외가 나면 DB는 롤백되지만 Redis 제거/이벤트 전송은 이미 수행되어 불일치가 생길 수 있다.
   - 근거: `src/main/java/com/kospot/multi/room/application/usecase/JoinGameRoomUseCase.java:42`, `src/main/java/com/kospot/multi/room/application/usecase/CreateGameRoomUseCase.java:38`

2. **room-not-found 분기 자체가 stale 상태를 유지시킬 수 있음**
   - leave에서 `member.leaveGameRoom()` 후 예외를 던지므로, 같은 트랜잭션 내에서는 롤백되어 `gameRoomId`가 다시 살아날 수 있다.
   - 근거: `src/main/java/com/kospot/multi/room/application/usecase/LeaveGameRoomUseCase.java:65`

3. **이벤트 발행 시점이 커밋 이전**
   - 현재 `@EventListener` 기반 즉시 처리라, 트랜잭션 최종 실패 시에도 알림이 먼저 나갈 수 있다.
   - 근거: `src/main/java/com/kospot/multi/room/application/usecase/LeaveGameRoomUseCase.java:97`, `src/main/java/com/kospot/multi/room/application/handler/GameRoomEventHandler.java:44`

### P1 (쿼리/부하 튐 가능성)

1. **`member.game_room_id` 인덱스 부재 가능성**
   - `findAllByGameRoomId`를 방 삭제/정합성 보정에서 반복 사용할 예정인데, 엔티티 인덱스 정의에 `game_room_id`가 없다.
   - 운영 데이터가 커지면 풀스캔 리스크가 높다.
   - 근거: `src/main/java/com/kospot/member/domain/entity/Member.java:21`, `src/main/java/com/kospot/member/infrastructure/persistence/MemberRepository.java:16`

2. **Redis `KEYS` 사용으로 순간 부하 가능성**
   - 현재 `getActiveRoomKeys()`가 `keys(pattern)` 사용 중이며, 키 수가 많아지면 Redis 단일 스레드 블로킹을 유발한다.
   - 근거: `src/main/java/com/kospot/multi/room/infrastructure/redis/service/GameRoomRedisService.java:354`

3. **join 경로 디버그 파일 write I/O**
   - `cannotJoinRoom`에서 매 요청 파일 append를 시도해 디스크 I/O와 지연 편차를 유발할 수 있다.
   - 근거: `src/main/java/com/kospot/multi/room/infrastructure/redis/service/GameRoomRedisService.java:323`

4. **브로드캐스터의 roomId/key 혼용 가능성**
   - `getActiveRoomKeys()`는 full Redis key를 반환하는데, 브로드캐스터는 이를 roomId로 가정해 재조합한다.
   - 키 재조합이 잘못되면 불필요한 Redis 조회 실패/재시도가 반복될 수 있다.
   - 근거: `src/main/java/com/kospot/multi/room/infrastructure/websocket/service/GameRoomPlayersBroadcaster.java:29`, `src/main/java/com/kospot/multi/room/infrastructure/redis/service/GameRoomRedisService.java:308`

1. **bulk update + clearAutomatically 부작용 가능성**
   - `updateHost`가 `clearAutomatically = true`라 영속성 컨텍스트가 비워지고, 같은 트랜잭션의 관리 엔티티 참조가 끊길 수 있다.

## 14) 반영할 안전장치(플랜 보강)

1. **트랜잭션 경계 분리**
   - `AutoLeaveRecoveryService`를 `REQUIRES_NEW`로 분리해 leave 정리를 독립 커밋
   - join/create 본 트랜잭션과 분리해 롤백 전파로 인한 정합성 깨짐 방지

2. **이벤트는 AFTER_COMMIT로 전환**
   - room leave/join 알림 계열을 `@TransactionalEventListener(phase = AFTER_COMMIT)`로 전환
   - "롤백됐는데 알림은 나간" 케이스 차단

3. **DB 인덱스 추가**
   - `member(game_room_id)` 단일 인덱스 추가
   - 필요 시 `(status, created_date)` 계열 인덱스도 조회 패턴 기반으로 검토

4. **Redis KEYS -> SCAN 전환**
   - 정합성 배치/브로드캐스터/cleanup에서 전부 SCAN cursor 방식 사용
   - 배치 크기 제한 + 실행 시간 상한 적용

5. **join 경로 파일 I/O 제거**
   - `cannotJoinRoom`의 debug file write 삭제(또는 feature flag + sampling)

6. **멱등 leave의 실패 코드는 제한적 유지**
   - 완화하되 `FATAL_FAILURE`, `RETRYABLE_FAILURE`를 명확히 분리
   - 성공성 완화로 장애 신호가 묻히지 않도록 메트릭/알람 임계치 분리

7. **브로드캐스터 입력 정규화**
   - `getActiveRoomKeys()`를 roomId 목록으로 바꾸거나, 브로드캐스터에서 key -> roomId 추출 후 사용

## 15) 순환 의존(Circular Dependency) 점검 결과

## 15-1. 현재 코드 정적 분석 결과

- `src/main/java` 기준 Spring Bean(`@Component/@Service/@Controller/@RestController/@UseCase/@Adaptor`) 의존 그래프를 정적 스캔했다.
- 대상: 273개 Bean, Bean 간 의존 엣지 514개
- 결과:
  - **순환 의존 0건**
  - **상호 양방향 직접 의존 pair 0건**

즉, **현재 시점에서는 생성자 주입 기준 순환 의존 문제는 확인되지 않았다.**

## 15-2. 리팩토링 시 새로 생길 수 있는 순환 의존 위험

이번 플랜(오케스트레이터/리컨실러 도입)에서 실수하기 쉬운 패턴은 다음이다.

1. **`RoomExitOrchestrator <-> LeaveGameRoomUseCase` 양방향 참조**
   - 오케스트레이터가 leave를 호출하는 구조는 가능
   - 반대로 leave가 오케스트레이터를 다시 주입받으면 즉시 순환

2. **`RoomMembershipReconciler <-> LeaveGameRoomUseCase` 양방향 참조**
   - reconciler가 멱등 leave를 호출하는 것은 가능
   - leave 내부에서 다시 reconciler를 호출하면 순환

3. **WS 핸들러/이벤트 핸들러의 역참조 추가**
   - `WebSocketEventHandler` -> orchestrator 단방향은 안전
   - orchestrator가 다시 `WebSocketEventHandler`를 참조하면 순환

## 15-3. 순환 의존 방지 원칙(필수)

1. **레이어 단방향 고정**
   - Handler/Controller -> Orchestrator -> Core UseCase -> Service/Adaptor
   - 역방향 참조 금지

2. **Leave를 "코어"로 고정**
   - `LeaveGameRoomUseCase`는 다른 상위 orchestration 컴포넌트를 주입받지 않음
   - 상위 컴포넌트가 leave를 호출하는 구조만 허용

3. **Reconciler는 조정자, Leave는 실행자**
   - Reconciler -> Leave 호출만 허용
   - Leave -> Reconciler 호출 금지

4. **이벤트는 단방향 전달로만 사용**
   - 동일 이벤트를 다시 유발하는 재귀 흐름 금지(이벤트 루프 방지)

## 15-4. 구현 체크리스트

- 신규 Bean 추가 시, 아래 3개 축으로 PR 체크:
  1) 생성자 주입 그래프에 역참조가 없는지
  2) 이벤트 핸들러가 호출 체인을 되감지 않는지
  3) `@Lazy`로 임시 봉합하지 않았는지 (`@Lazy`는 금지, 구조로 해결)
