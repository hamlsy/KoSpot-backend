# 게임방 유령 방 및 퇴장 흐름 정합성 이슈

## 이슈 요약

- 게임방 퇴장 후 실제 인원이 0명인데 방이 남는 유령 방이 간헐적으로 발생한다.
- 유저의 `gameRoomId` 상태가 정리되지 않아 다른 방 입장/새 방 생성이 막히는 케이스가 있다.
- `leave -> join/create` 전환 시 자동 퇴장 보정이 실패하면 전체 흐름이 중단된다.
- WebSocket disconnect/unsubscribe/화면 이동(`navigate-room`) 경로에서 퇴장 처리 일관성이 부족하다.

## 문제 현상 (재현 관측)

1. 게임방에서 나갔는데 방이 남아 있음 (아무도 없는데 삭제 안 됨)
2. 그 상태에서 다른 방 입장이 안 됨 (이전 방 자동 퇴장 실패)
3. 그 상태에서 새 방 생성이 안 됨 (이전 방 자동 퇴장 실패)
4. 결과적으로 유령 방이 누적되고, 일부 세션에서 퇴장 불가 상태가 지속됨

## 기대 동작 (정의)

- 퇴장 요청은 멱등(idempotent)해야 한다.
  - 이미 나간 상태/방 없음/Redis 플레이어 없음이어도 "정리 완료"로 처리되어야 한다.
- `join`/`create`는 이전 방 상태가 꼬여 있어도 자동 보정 후 성공 가능해야 한다.
- DB(`member.gameRoomId`)와 Redis(`game:room:{id}:players`) 불일치가 남지 않아야 한다.
- 마지막 인원 퇴장 시 방 데이터는 즉시 정리되어야 한다.

## 현재 흐름 분석 포인트

### 1) HTTP 흐름

- `CreateGameRoomUseCase`에서 `autoLeaveIfInRoom` 호출 후 생성 수행
- `JoinGameRoomUseCase`에서 `autoLeaveIfInOtherRoom` 호출 후 입장 수행
- 두 경로 모두 선행 leave 실패 시 후속 동작이 막히는 구조

### 2) Leave 핵심 흐름

- `LeaveGameRoomUseCase`는 Redis 기반 제거 + DB 반영 + 이벤트 발행 구조
- 방/플레이어 미존재 케이스를 예외로 간주하여 흐름을 실패 처리
- 이 예외가 상위 `join/create` 자동 퇴장 경로까지 전파되어 사용자 액션을 차단

### 3) WebSocket 흐름

- disconnect 시 grace period 후 이벤트 기반 leave 처리 경로 존재
- unsubscribe 기반 게임방 leave 경로는 비활성/보완 필요
- `navigate-room` 이유로 skip 처리되는 케이스에서 후속 진입 성공 시 이전 방 정리 보장 로직이 약함

## 가설 원인

1. **Leave 실패 기준이 과도함**
   - "정합성 불일치"를 복구 가능한 상태가 아닌 즉시 에러로 처리
2. **자동 퇴장 의존 흐름**
   - `join/create`가 auto-leave의 성공을 강하게 가정
3. **DB/Redis 소스 불일치 허용**
   - Redis TTL/세션 종료 타이밍/이벤트 지연으로 한쪽만 정리되는 창이 존재
4. **이탈 경로 다중화에 대한 일관성 부족**
   - HTTP leave, WS disconnect, navigate 전환이 동일한 정리 규칙으로 수렴하지 않음

## 해결 목표

### Goal A. 퇴장 멱등화

- `LeaveGameRoomUseCase`를 "반드시 정리" 중심으로 재설계
- 이미 퇴장된 상태는 에러가 아니라 성공성 결과(`ALREADY_LEFT`)로 처리

### Goal B. 입장/생성 자동 보정 강화

- `join/create` 선행 auto-leave 실패를 흡수하고 정합화 후 진행
- 사용자 체감 기준으로 "다른 방 입장 불가/새 방 생성 불가" 제거

### Goal C. 정합성 자동 복구

- 요청 진입 시점 경량 reconcile + 배치성 orphan cleanup 도입
- 유령 방/댕글링 멤버를 주기적으로 제거

### Goal D. WS 이탈 경로 통합

- unsubscribe/disconnect/navigate 경로를 동일한 정리 정책으로 수렴
- grace period 종료 후 leave 처리 실패율 및 누락률 최소화

## 작업 범위 (초안)

1. Leave 결과 모델 확장 (성공/이미정리/재시도불가 구분)
2. `LeaveGameRoomUseCase` 멱등 보장 리팩토링
3. `JoinGameRoomUseCase` / `CreateGameRoomUseCase` 자동 보정 로직 강화
4. WebSocket 이벤트 핸들러의 room leave 트리거 일관화
5. 정합성 점검/정리 스케줄러 추가 (옵션: feature flag)
6. 테스트 추가
   - stale `gameRoomId` 상태에서 join/create 성공
   - Redis 플레이어 없음 상태 leave 성공 처리
   - 마지막 인원 leave 시 room 삭제 보장
   - disconnect/navigate 전환 시 유령 방 미발생

## 수용 기준 (Acceptance Criteria)

- [ ] 마지막 인원 퇴장 후 유령 방이 남지 않는다.
- [ ] 이전 방 상태가 꼬여 있어도 다른 방 입장/새 방 생성이 성공한다.
- [ ] leave API/이벤트는 멱등하게 동작한다.
- [ ] DB/Redis 불일치 케이스가 자동 정리된다.
- [ ] 회귀 테스트(동시성 포함) 통과 및 재현 시나리오에서 문제 미재발.

## 측정 지표

- `leave_already_left_count`
- `auto_leave_recovery_count`
- `ghost_room_cleaned_count`
- `join_blocked_by_stale_room_count` (최종 0 목표)

## 리스크 및 대응

- 리스크: leave를 성공성으로 완화하면서 실제 장애 신호가 묻힐 수 있음
  - 대응: 결과 코드를 세분화하고 경고 로그 + 메트릭으로 분리 관측
- 리스크: reconcile 배치가 과도하면 Redis/DB 부하 증가
  - 대응: 대상 범위 제한, 샘플링, 점진 배포

## 우선순위

1. Leave 멱등화 + join/create 자동 보정
2. WS 이탈 경로 일관화
3. 정합성 배치/메트릭 고도화

## 비고

- 본 이슈는 "기능 추가"보다 "정합성/복원력 강화" 성격의 안정화 이슈다.
- 구현 시 기존 분산 락 전략(Lua/기타)의 보장 범위를 유지한 상태로 보완한다.
