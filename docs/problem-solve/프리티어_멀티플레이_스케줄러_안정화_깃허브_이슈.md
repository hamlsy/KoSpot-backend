# [Issue] 프리티어 환경 멀티플레이 스케줄러/타이머 안정화

## 제목

프리티어(1 vCPU/1GB) 환경에서 멀티플레이 스케줄러 부하로 인한 OOM/지연 위험 완화

## 배경

- 멀티플레이 라운드 진행 중 타이머/전환/로딩 타임아웃 스케줄링이 동시에 발생한다.
- 현재 설정은 일반 환경에서는 동작하지만, 프리티어 리소스에서는 스레드/메모리 여유가 작아 장애 여지가 크다.
- 저장소 내 `hs_err_pid*.log`가 다수 존재하며, 네이티브 메모리 부족(OOM) 흔적이 확인된다.

## 문제 요약

1. JVM 메모리 여유 부족
- `Dockerfile`에서 `-XX:MaxRAMPercentage=75.0` 사용 중.
- 1GB급 인스턴스에서 힙 비중이 높아 네이티브 메모리(스레드 스택/직렬화/네트워크 버퍼) 부족 위험이 큼.

2. 스케줄러/비동기 스레드 자원 과다 사용 가능성
- 타이머/플로우/WS 인바운드/아웃바운드/채팅 실행기가 동시에 활성화됨.
- `@Async` 기본 실행기 경로가 명시적이지 않아 피크 시 포화/지연 추적이 어려움.

3. 취소 태스크 정리 정책 불일치
- 일부 스케줄러는 `removeOnCancelPolicy(true)`가 적용되어 있으나, 게임 타이머/플로우 스케줄러에는 누락됨.
- 취소가 많은 시나리오(라운드 조기종료/게임 취소)에서 큐 잔존 비용 발생 가능.

4. 주기성 브로드캐스트의 상시 스캔 비용
- 10초마다 모든 활성 룸 대상 브로드캐스트 수행.
- 활성 룸 수 증가 시 Redis 조회 + 메시지 전송이 고정 주기로 누적.

5. 정합성 cleanup 활성화 시 부하 급증 여지
- cleanup은 feature flag로 보호되지만, 활성화 시 Redis/DB를 주기적으로 순회.
- 프리티어에서 배치 크기/주기 튜닝 없이 켜면 CPU/DB 부하가 튈 가능성이 큼.

6. 관측 지표 부족
- 스케줄러 큐 길이/지연/거부율/실행시간에 대한 메트릭이 부족해 사전 탐지가 어려움.

## 목표

- 프리티어에서도 멀티플레이 라운드 진행 시 서버가 재시작/OOM 없이 안정 동작.
- 타이머 정확도(5초 sync)와 게임 진행 정합성은 유지.
- 병목이 생겨도 즉시 관측 가능하도록 메트릭/로그를 보강.

## 범위

### P0 (이번 이슈에서 반드시)

- JVM 메모리 가드레일 도입
  - 프리티어 기본값 재조정(예: `MaxRAMPercentage` 하향, 필요 시 `-Xss`/direct memory 상한 명시)
  - 환경변수로 안전하게 override 가능하게 구성

- 스케줄러/실행기 튜닝
  - `gameTimerTaskScheduler`, `gameFlowTaskScheduler`에 `removeOnCancelPolicy(true)` 적용
  - 프리티어 기준 thread pool 기본값 재산정 및 설정 외부화
  - `@Async` 기본 실행기 경로를 명시적으로 고정

- 주기 브로드캐스트 부하 완화
  - 룸 전체 브로드캐스트 주기 configurable화
  - `parallelStream()` 사용 여부 재검토(작은 코어 환경 최적화)
  - 활성 룸 수가 많을 때 배치/샘플링/스킵 정책 중 1개 이상 적용

- cleanup 안전 장치
  - cleanup 활성화 시 타임버짓/배치 상한/로그 샘플링 적용
  - 운영 기본값은 보수적으로 유지(비활성 또는 저주기)

### P1 (가능하면 포함)

- 프리티어 전용 프로파일(`application-freetier.yml`) 추가
- 피크 시 graceful degradation 정책(예: 비필수 브로드캐스트 빈도 자동 완화)

### P2 (후속)

- 이벤트 기반 플레이어 목록 갱신 전환(폴링 최소화)
- 스케줄러 상태 대시보드(Grafana/Prometheus) 정식 구축

## 작업 체크리스트

- [ ] JVM 옵션 재설계 및 문서화
- [ ] Async/Scheduler 풀 크기 재설정 및 설정 외부화
- [ ] 게임 타이머/플로우 스케줄러 `removeOnCancelPolicy(true)` 반영
- [ ] 룸 플레이어 브로드캐스터 부하 완화(주기/병렬화/배치)
- [ ] cleanup 스케줄러 안전 장치 보강
- [ ] 메트릭 추가(큐 길이, active thread, task latency, reject count)
- [ ] k6 재시험 및 결과 문서 업데이트

## 완료 조건 (Acceptance Criteria)

1. 안정성
- 400 VU 타이머 시나리오 10분 이상 구동 시 프로세스 다운/OOM 없음
- 테스트 동안 `hs_err_pid*.log` 신규 생성 없음

2. 성능/정확도
- 타이머 간격 `p95 < 5500ms`
- interval accuracy `>= 95%`
- 피크 구간에서도 라운드 전환 누락/중복 시작 없음

3. 운영성
- 스케줄러/실행기 상태를 로그/메트릭으로 추적 가능
- 설정값만으로 프리티어/일반 환경 스위칭 가능

## 비범위 (Out of Scope)

- 게임 규칙/점수 로직 변경
- 팀전 도메인 기능 확장
- Redis/DB 인프라 자체 교체

## 검증 시나리오

- `k6-scripts/timer-interval-test.js` 실행(최대 400 VU)
- 룸 수 증가(10 -> 25 -> 50) 단계별 CPU/스레드/메모리 추적
- 조기 종료/라운드 취소/방 삭제가 섞인 혼합 시나리오에서 태스크 누수 점검
