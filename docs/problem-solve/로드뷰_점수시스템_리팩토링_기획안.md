# KOSPOT 점수 시스템 상세 기획안 (ScoreCalculator 리팩토링)

## 1) 기획 목표 (Objective)

- **정확도 우선**: 로드뷰 위치 추적 게임의 본질인 거리 오차 최소화를 핵심 실력 지표로 삼는다.
- **긴장감 부여**: 동점/유사 오차 구간에서 변별력을 위해 시간(스피드) 요소를 가중치로 반영한다.
- **긍정적 UX**: 감점 방식이 아닌 보너스 배율 방식으로, 늦게 맞춰도 기본 점수는 보존한다.
- **모드 호환성**: 제한시간이 다른 싱글/멀티 모드에서도 추가 분기 없이 동작하는 점수 체계를 채택한다.

---

## 2) 현재 코드 진단 (현행 `ScoreCalculator` 기준)

대상 파일: `src/main/java/com/kospot/game/common/utils/ScoreCalculator.java`

### 현재 상태

- 거리 구간별 선형 보간 점수는 이미 구현되어 있음.
- 멀티 점수는 `distance score * 0.08`로 단순 환산.
- 시간 기반 배율(Grace/Decay)은 미구현.

### 확인된 개선 포인트

1. **시간 요소 부재**
   - 현행 시그니처는 거리만 입력받아 계산 (`calculateScore(distance)`).
   - 기획안의 핵심인 `Time Multiplier` 반영 불가.

2. **유지보수성 낮음**
   - 거리 구간이 하드코딩된 `if-else` 체인.
   - 구간 추가/수정 시 코드 수정 범위가 넓고 실수 여지 큼.

3. **상수 정합성 이슈**
   - `MAX_SCORE`는 1000인데, `MAX_SCORE_BD`는 10000으로 설정되어 있음.
   - 현재 로직상 큰 장애는 아니지만, 설계 의도와 불일치.

4. **반올림 정책 불명확**
   - 점수 반환 타입은 `double`이고 소수 2자리 반올림.
   - 랭킹/정산에서는 결국 `int` 정수화가 필요하므로 정책을 명시적으로 통일할 필요가 있음.

5. **도메인 연결 정보 분산**
   - 싱글: `RoadViewGame`에서 거리 점수 사용.
   - 멀티: `RoadViewSubmission`에서 거리 점수 사용.
   - 시간 값(`answerTime`, `timeToAnswer`)은 존재하지만 점수 계산기로 전달되지 않음.

## 3) 핵심 점수 구조 (Core Mechanics)

최종 점수:

```text
FinalScore = BaseScore(distanceKm) * TimeMultiplier(elapsedMs, limitMs, graceMs)
```

- 최대 점수: **1300점** (거리 0km + 유예 시간 내 제출)
- 최소 점수: **0점** (거리 400km 이상)

---

## 4) 거리 기반 기본 점수 (Base Score)

거리 기준점(단위: km)과 목표 점수:

- 0 -> 1000
- 1 -> 950
- 5 -> 900
- 10 -> 800
- 30 -> 700
- 50 -> 600
- 100 -> 400
- 200 -> 200
- 400 이상 -> 0

구간 내 계산은 선형 보간:

```text
t = (x - x1) / (x2 - x1)
score = y1 + t * (y2 - y1)
```

---

## 5) 시간 가중치 시스템 (Time Multiplier)

### 5.1 유예 시간 (Grace Period)

- 기본값: 5초 (5000ms)
- `elapsed <= grace` 이면 무조건 `1.3` 적용

### 5.2 유예 이후 선형 감소 (Linear Decay)

```text
multiplier = 1.0 + 0.3 * (limitMs - elapsedMs) / (limitMs - graceMs)
```

- 범위 클램프: `[1.0, 1.3]`
- 제한시간 직전/종료 시점에도 최소 `1.0` 보장

---

## 6) 게임 모드별 동적 호환

- **싱글 랭크**: 제한시간은 **3분(180초) 고정**이며, 현재 클라이언트 타이머 기준으로 `answerTime`이 전달된다.
- **멀티**: 별도 모드 분기 없이, `MultiRoadViewGame.timeLimit` 값을 그대로 사용해 선형 감소.
- 배율은 절대시간 하드코딩이 아니라 `limitMs` 기반 비율 계산이므로, 멀티 룸 설정 시간 변경에도 추가 코드 수정이 불필요.

멀티 시간 제한값 출처:

- `src/main/java/com/kospot/multi/game/domain/entity/MultiRoadViewGame.java`
- 필드: `timeLimit`

시간 제한 전달 경로(현재 코드 기준):

- `RoadViewGameRoundService.createGameRound(...)`에서 `game.getTimeLimit()`을 `RoadViewGameRound.createRound(...)`로 주입
- `RoadViewGameRound`는 `BaseGameRound.timeLimit`에 저장되고, `getDuration()`으로 ms 환산 가능

멀티 최종 환산:

```text
multiFinal = finalScore * 0.08
```

## 7) 백엔드 리팩토링 설계

## 7.1 `ScoreCalculator` API 재구성

기존:

- `calculateScore(double distance)`
- `calculateMultiGameScore(double distance)`

변경 제안:

- `calculateBaseScore(double distanceKm)`
- `calculateTimeMultiplier(long elapsedMs, long limitMs, long graceMs)`
- `calculateFinalScore(double distanceKm, long elapsedMs, long limitMs, long graceMs)`
- `calculateFinalMultiScore(double distanceKm, long elapsedMs, long limitMs, long graceMs)`

호환성 유지를 위해 과도기에는 기존 메서드를 `@Deprecated` 처리하고 신규 메서드 내부 위임을 권장.

## 7.2 거리 테이블 구조화 (TreeMap)

`if-else` 제거, `TreeMap<Double, Double>` 기반으로 다음/이전 기준점을 탐색:

- `floorEntry(distance)`
- `ceilingEntry(distance)`

복잡도: `O(log N)`

장점:

- 구간 추가/변경이 데이터 선언 수정으로 끝남
- OCP(개방-폐쇄 원칙) 충족

## 7.3 반올림/정수화 정책 통일

- 내부 계산은 `double`
- 반환 직전 `BigDecimal.setScale(0, HALF_UP)`로 정수화 (`int`)
- API/DB 저장 스펙을 정수 중심으로 단일화

> 단, 기존 응답이 소수점 점수에 의존 중이라면 단계적 이행 필요(아래 9절).

## 7.4 설정값 외부화

다음 값들은 하드코딩 대신 설정으로 분리 권장:

- grace 시간(기본 5초)
- 최대 보너스(기본 1.3)
- 멀티 환산 배수(기본 0.08)
- 거리-점수 기준표

## 8) 도메인/유즈케이스 반영 포인트

### 싱글

- 현재 종료 경로:
  - `EndRoadViewRankUseCase.execute(...)`
  - `RoadViewGameService.finishGame(...)`
  - `RoadViewGame.end(member, submittedLat, submittedLng, answerTime, distance)`
- 현재 문제:
  - `answerTime`가 전달되지만 `RoadViewGame.getScore(distance)`에서 시간 배율을 사용하지 않음
- 변경안:
  - `RoadViewGame.end(...)` 내부에서 `ScoreCalculator.calculateFinalScore(distance, elapsedMs, limitMs, graceMs)` 호출
  - 싱글 랭크 제한시간은 `SINGLE_RANK_LIMIT_MS = 180_000` 고정 상수로 사용
  - `answerTime`는 클라이언트 측 경과시간을 수신하므로, 서버에서 `0 <= answerTime <= 180_000` 범위 클램프 후 계산
- 대상:
  - `src/main/java/com/kospot/game/application/usecase/rank/EndRoadViewRankUseCase.java`
  - `src/main/java/com/kospot/game/application/service/RoadViewGameService.java`
  - `src/main/java/com/kospot/game/domain/entity/RoadViewGame.java`
  - `src/main/java/com/kospot/game/presentation/dto/request/EndGameRequest.java`

단위 정합성 체크(싱글):

- `EndGameRequest.RoadView.answerTime` 단위(ms/sec) 문서화 필요
- 멀티는 `timeToAnswer // 밀리초`로 명시돼 있으므로, 싱글도 ms로 통일 권장
- 클라이언트 제공값 신뢰 최소화: 서버 기준 게임 시작/종료 시각과 허용 오차(예: 1~2초) 검증 로직을 2차 단계로 추가 권장

### 멀티

- 현재 제출/점수 경로:
  - `SubmitRoadViewPlayerAnswerUseCase.execute(...)`
  - `RoadViewSubmissionService.createPlayerSubmission(...)`
  - `RoadViewSubmission.assignDistanceAndPlayerScore(coordinate)`
  - 이후 `EndRoadViewSoloRoundUseCase.execute(...)`에서 정렬/누적 처리
- 현재 문제:
  - `timeToAnswer`는 저장되지만 점수 계산에 `timeLimit`이 반영되지 않음
  - `EndRoadViewSoloRoundUseCase`는 라운드 종료/정렬만 담당하므로 점수 계산 시점은 제출 저장 시점이 적합
- 변경안(권장):
  - 점수 계산 책임을 `RoadViewSubmissionService.createPlayerSubmission(...)`로 집중
  - `round.getDuration().toMillis()`를 `limitMs`로 사용
  - `submission.getTimeToAnswer()`를 `elapsedMs`로 변환해 `calculateFinalMultiScore(...)` 호출
  - `RoadViewSubmission` 엔티티의 점수 계산 메서드는 최소화(거리 계산/세터 정도로 축소)
- 대상:
  - `src/main/java/com/kospot/multi/submission/application/usecase/SubmitRoadViewPlayerAnswerUseCase.java`
  - `src/main/java/com/kospot/multi/submission/application/service/RoadViewSubmissionService.java`
  - `src/main/java/com/kospot/multi/submission/entity/roadview/RoadViewSubmission.java`
  - `src/main/java/com/kospot/multi/round/application/usecase/roadview/solo/EndRoadViewSoloRoundUseCase.java`

정렬 정책 유지 여부:

- 현재 멀티 정렬은 `distance ASC -> timeToAnswer ASC`
- 시간 배율이 점수에 반영되더라도, 동점 안정성과 UX 일관성을 위해 기존 tie-break 유지 권장

### 테스트 엔드포인트

- `/roadView/scoreTest/{distance}`는 시간 인자 없는 구 API
- 신규 검증용으로 쿼리 파라미터 기반 확장 필요
  - 예: `/roadView/scoreTest?distance=...&elapsedMs=...&limitMs=...`
- 대상: `src/main/java/com/kospot/game/presentation/controller/RoadViewGameController.java`

## 9) 단계별 적용 계획 (Rollout Plan)

### Phase 1: 계산기 리팩토링 (기능 추가 + 역호환)

- `ScoreCalculator` 내부를 테이블/배율 구조로 개편
- 신규 메서드 추가, 기존 메서드는 유지(레거시 경로 보호)
- 단위 테스트 추가

### Phase 2: 싱글 점수 경로 이행

- `EndRoadViewRankUseCase -> RoadViewGameService -> RoadViewGame.end` 체인에서 시간 인자 기반 계산으로 전환
- 싱글 한정 `limitMs=180_000` 고정 적용
- `answerTime` 단위를 ms로 고정(요청 DTO/프론트 스펙 동시 업데이트)
- 서버에서 `answerTime` 범위 검증/클램프 적용
- 실게임 로그 비교(변경 전/후)로 분포 확인

### Phase 3: 멀티 점수 경로 이행

- `RoadViewSubmissionService.createPlayerSubmission`에서 `round.getDuration().toMillis()` 기반 계산으로 전환
- `EndRoadViewSoloRoundUseCase`는 랭킹/누적 전용으로 역할 고정(점수 재계산 금지)
- 멀티 환산(0.08) 포함한 회귀 테스트

### Phase 4: 문서/스펙 확정 및 레거시 제거

- API 문서, 운영 가이드 업데이트
- 필요 시 구 메서드 제거

## 10) 테스트 전략

## 10.1 단위 테스트

- 거리 경계값: `0, 1, 5, 10, 30, 50, 100, 200, 400`
- 시간 경계값: `0, grace, grace+1ms, limit-1ms, limit, limit+1ms`
- 음수/비정상 입력: 방어 로직 확인

## 10.2 속성 기반 테스트

- 거리 증가 시 기본 점수 비증가(monotonic)
- 동일 거리에서 시간 증가 시 최종 점수 비증가
- multiplier 범위 `[1.0, 1.3]` 보장

## 10.3 회귀 테스트

- 멀티 기존 랭킹 정렬 영향 점검
- 싱글 최고점/평균점 추이 모니터링

## 11) 프론트/기획 공유 포인트

- 동점 처리: `distance -> timeToAnswer` 우선순위 정렬 정책을 명시
